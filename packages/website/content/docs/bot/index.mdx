OpenKitten Bot is a self-hosted AI assistant that lives inside your Telegram account. You chat with it like any other contact — it streams replies, handles files, runs tools, and remembers each conversation per chat or forum topic.

Under the hood it wires [grammY](https://grammy.dev) (Telegram) to [OpenCode](https://opencode.ai) (agent runtime), so you inherit OpenCode's full ecosystem of providers, agents, commands, skills, and MCP servers — all driven from a phone-friendly chat UI.

## The bot extends itself

Two built-in skills turn chat into a live authoring surface:

- **`custom-commands`** — say *"add a `/weather` command that takes a city and replies with the forecast"*. The skill writes `~/.openkitten/profiles/<profile>/.opencode/commands/weather.md` for you, then calls the `reload_commands` MCP tool. Telegram's command menu updates immediately; **no restart required**.
- **`skill-creator`** — say *"teach yourself how to work with our expense tracker"*. The skill scaffolds a new `SKILL.md` (or installs one from [skills.sh](https://skills.sh) via `npx skills add …`), then calls `reload_extensions`. OpenKitten sends a ⏳ notification, restarts its OpenCode server to pick up the new skill, and sends back a ✅ confirmation when the skill is active.

Both flows are driven entirely from inside Telegram — no shell, no editor, no SSH.

## How it runs

`bun . serve` boots a single process that owns four cooperating pieces:

1. A grammY bot connected to the Telegram long-polling API.
2. An embedded OpenCode server running your configured providers and agents.
3. A local MCP server that exposes OpenKitten-specific tools (file sending, schedules, reloads) back into OpenCode.
4. A SQLite database and a Bunqueue-backed scheduler for conversation state and cron jobs.

Each Telegram chat — or each forum topic in a group — maps to exactly one OpenCode session. Typing indicators, message-edit streaming, and per-chat lock/pending semantics are all handled transparently.

## Private by default

Only the configured Telegram `userId` can talk to the bot. Every inbound update is filtered before it reaches any handler. The bot token and user ID are stored with file mode `0600` under your profile directory, never committed to the repo.
