title: Schedules
description: Recurring and one-time tasks that fire on a cron pattern and report back to your chat.
---

OpenKitten has a built-in scheduler that runs tasks on a cron expression. Each tick runs in a fresh isolated OpenCode session, the agent produces a final text response, and the result is either delivered to your Telegram chat or stays silent — based on what the task itself says.

You manage schedules by talking to the bot in natural language. The agent has the **`schedules`** skill active by default, so it knows how to use the right tools.

## Creating a schedule

Just ask. Some examples:

- *"Every day at 9am, summarize yesterday's commits across my repos."*
- *"Check my Gmail for new bank emails every hour. If any, list them for me to classify."*
- *"Remind me in 30 minutes to move my laundry."*
- *"Every Monday morning, send me my calendar for the week."*

The bot will:

1. Check the server time.
2. Pick a timezone (asks if unclear — otherwise uses UTC).
3. Compose a cron expression.
4. Create the schedule and confirm the ID back.

You don't have to memorize cron syntax or IDs — the agent handles both.

## What happens when a schedule fires

Each scheduled tick produces one **run** with a terminal status:

<Cards>
  <Card title="reported" description="The task produced meaningful output. The text is delivered to your Telegram chat as a message." />
  <Card title="silent" description="The task explicitly said there's nothing to report (or produced no text). Nothing is sent to your chat — but the run is still recorded in history." />
  <Card title="failed" description="The task threw an error. Recorded with the error message. By default stays silent; set notifyOnFailure to get a chat alert." />
  <Card title="cancelled" description="You explicitly cancelled the run, or the schedule was deleted mid-execution, or an overlap policy preempted it." />
  <Card title="skipped" description="A cron tick was dropped because an earlier run for this schedule was still active (only with overlap: skip)." />
</Cards>

**Silent-by-default is a feature, not a bug.** It's what makes "check Gmail every 5 minutes" usable without 288 notifications a day. The task only speaks when there's something real to say.

## Inspecting history

Every run — reported, silent, failed, cancelled, skipped — is persisted. Ask the agent:

- *"Show me the last 10 runs of the Gmail check."*
- *"Did any run fail today?"*
- *"What did the 9am summary say on Tuesday?"*
- *"Show me the full output of that one run."*

The agent uses `queue_runs` and `queue_run_get` under the hood. You never see the tools — only the answer.

## Managing schedules

| You say | The bot does |
|---|---|
| *"What schedules do I have?"* | lists all schedules with status |
| *"Pause the Gmail check for the weekend."* | disables without deleting (history preserved) |
| *"Turn it back on."* | re-enables; next cron tick resumes |
| *"Change it to every 2 hours instead."* | updates cron, re-registers atomically |
| *"Run the report now, don't wait."* | triggers immediately, regardless of cron |
| *"Cancel the one that's currently running."* | aborts the active run |
| *"Delete it."* | removes permanently; all run history cascade-deletes |

Pause (disable) and delete are **different**. Disable is reversible and keeps history. Delete is destructive.

## Options you can ask for

When creating or updating a schedule, the agent understands these concepts:

### Timezone

Any IANA zone: `Asia/Ho_Chi_Minh`, `Europe/London`, `America/New_York`, `UTC` (default).

*"Run at 9am Vietnam time."* → cron fires at 09:00 `Asia/Ho_Chi_Minh`.

### once

Fires exactly once at the next matching cron time, then auto-pauses. Row is preserved so you can see it in history.

*"Remind me once in 2 hours to call mom."* → cron `<+2hr>`, `once: true`.

### overlap

What happens if a new tick fires while the previous run is still executing:

- **queue** (default) — let them stack.
- **skip** — drop the new tick. Good for expensive tasks that should never overlap.
- **cancel_previous** — abort the active one, run the new one. Good for "always snapshot current state" patterns.

### notifyOnFailure

Default is silent-on-failure (failures recorded but not shown in chat). Set to true for critical tasks where a silent failure is dangerous.

*"If my deployment-health check ever fails, tell me immediately."* → `notifyOnFailure: true`.

### maxRuntimeMs

Cap how long the ephemeral agent gets before the run is finalized as `silent`. Default 15 minutes. Useful for fast checks where a hang would be noise:

*"Quick ping every minute — give up after 10 seconds."* → `maxRuntimeMs: 10000`.

## Cron reference

OpenKitten uses standard 5-field cron: `minute hour day-of-month month day-of-week`.

| Expression | Meaning |
|---|---|
| `0 9 * * *` | every day at 09:00 |
| `*/15 * * * *` | every 15 minutes |
| `0 9 * * 1-5` | weekdays at 09:00 |
| `0 0 1 * *` | midnight on the first of each month |
| `@hourly` | top of every hour |
| `@daily` | midnight every day |
| `@weekly` / `@monthly` | shortcut aliases |

All expressions are interpreted in the schedule's timezone (default `UTC`).

## Prompt wording tips

When you describe a task, the agent turns your words into the schedule's prompt. The prompt runs in a fresh ephemeral session each time — it doesn't carry any memory of your chat. A few things that help:

**For tasks that often have nothing to report**, include the silent-contract. The agent does this automatically when you say things like "only tell me if …" or "check silently":

> *If there is nothing to report, respond with exactly `[NO_REPORT]`. Otherwise, reply with a concise summary.*

**For reminders** — just phrase it as the text you want to hear:

> *"Remind me to take my vitamins."* — the scheduled agent will say "Remind you to take your vitamins" at the fire time.

**For multi-step checks** (API calls, multiple tool uses), be explicit about what to do with each outcome:

> *"Check the deployment pipeline. If green, `[NO_REPORT]`. If red, list the failing stage and the error."*

## Troubleshooting

If a schedule looks off, ask the bot:

1. *"What's the server time?"* — confirms clock and resolved timezone.
2. *"Is the Gmail schedule still enabled?"* — confirms it exists and isn't paused.
3. *"Show me the last 5 runs."* — reveals whether anything has fired and what the statuses were.
4. *"Get full details of run `<id>`."* — output + error for a specific run.

If *none of this* helps, the agent can inspect the low-level queue via `queue_status` / `queue_list_crons` / `queue_list_jobs`, but that's rarely needed — the run history almost always has the answer.

## Full MCP tool list

The agent uses these tools behind the scenes. You don't call them directly, but they're listed here for reference.

> **Naming:** each tool is registered on OpenKitten's MCP server under the unprefixed name below (`queue_schedule_create`, etc.). OpenCode exposes them to agents with the server name prepended (`openkitten_queue_schedule_create`). Skill authors write the prefixed form; this page shows the unprefixed form.


### Schedules

| Tool | Purpose |
|---|---|
| `queue_schedule_create` | create a schedule |
| `queue_schedule_list` | list schedules (filter by session or enabled flag) |
| `queue_schedule_update` | change any field |
| `queue_schedule_delete` | remove permanently (cascade-deletes run history) |
| `queue_schedule_enable` / `_disable` | resume / pause without losing history |
| `queue_schedule_trigger` | fire immediately, ignoring cron and overlap; returns the new `runId` directly |

### Runs (history)

| Tool | Purpose |
|---|---|
| `queue_runs` | list runs with filters (`scheduleId`, `sessionId`, `status`, `trigger`, `since`, `until`, `limit`, `offset`) |
| `queue_run_get` | fetch one run by id, full output / error |
| `queue_run_cancel` | abort an in-flight run |

### Meta

| Tool | Purpose |
|---|---|
| `queue_server_time` | current time in UTC / local, useful before computing cron |

### Diagnostic (rarely needed)

| Tool | Purpose |
|---|---|
| `queue_status` | queue paused state + raw job counts |
| `queue_list_jobs` | raw Bunqueue job listing by state |
| `queue_list_crons` | registered cron schedulers |

## Hygiene

- Delete test schedules you don't need anymore — disable isn't enough if you won't re-use them.
- Run history for a schedule is capped at the most recent 500 runs; older ones are pruned automatically. For high-frequency schedules, pull important output into your own store or ask the agent to archive it.
- If the bot restarts mid-run, any in-flight run is finalized as `failed` with `error: "bot restart"`. The next cron tick runs fresh. This is normal.
